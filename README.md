### Роли и права:
1. `ADMIN` - авторaизация => вход в админ-панель, внесение данных пациентов.
    - Импорт и экспорт данных:
        - может загрузить данные из Excel
        - может выгрузить данные в Excel
    - Возможно какие-то дашборды в будущем
    - Обучение модели
2. `PATIENT` - обычный посетитель сайта, без авторизации (анонимный пользователь), проходит опрос и получает результат.

Те же сущности, но что стоит за полями:
- `Glossary` – словарь:
    - `Name` – название словаря.

- `Glossary value` – значения словарь:
    - `Value` – значение словаря,
    - `Glossary_id` – ссылка на словарь.

- `Patient` - то же, что описано выше в структуре запрашиваемых данных

- `Diagnosis` – таблица с диагнозами пациентов:
    - `Patient Id` – номер пациента,
    - `Diagnosis_id` – ссылка на диагноз из словаря,
    - `Year of diagnosis` – год постановки диагноза.

### Ключевые бизнес-правила:

```
1. Данные опроса сохраняются атомарно: Либо все поля `Patient` и `Diagnosis` записываются, либо ни одно
2. Импорт Excel выполняется транзакционно - при ошибке вся загрузка откатывается или пропускается с логированием
3. Администратор не може получить доступ к панели без действительного JWT-токена
4. Год диагноза не может быь в будущем и не раньше 1900 года
5. Пациент не может отправить форму с незаполненными обязательными полями
```

### Страницы:
```
- Приветственная страница (окошко), выбор `я здесь чтобы пройти опрос` / `я администратор`. Уже после этого переводит на страницу авторизации либо на страницу опросника.
    ├── 4 страницы опросника (возможно изменится)
        └── страница окончания опроса с результатами (вывод => с такой-то вероятностью вы     получите сердечно-сосудистое заболевание ...)
    ├── Страница авторизации
        └── Страница администратора (корректная пара логин-пароль)
            ├── Загрузка данных (импорт из Excel-таблиц)
            └── Выгрузка данных (экспорт в Excel-таблицы)
```

Возможно подключение дашбордов в дальнейшем.

### Обязательные пользовательские сценарии (MVP):
- Пациент проходит опрос и получает результат => результат опроса сохраняется в БД
- Администратор авторизовывается
- Администратор может посмотреть статистику по пациентам и их данные (данные из результатов опроса)
    - (в виде какого-то дашборда, возможно)
- Администратор может импортировать/экспортировать данные в формате Excel-таблиц

### Минимальный API-контур:
- API для связки с БД (точный стек неопределён со стороны другой команды, ждём информации)
- API для связки с ИИ для анализа данных и прогнозирования заболеваний

```
# Публичные эндпоинты (пока не понятно, обычные пользователи без авторизации)
1. GET  /api/glossaries                   — получить все словари (Glossary + GlossaryValue) для отрисовки форм опросника
2. GET  /api/glossaries/{id}/values       — получить значения конкретного словаря
3. POST /api/survey                       — отправить заполненный опрос пациента (создаёт Patient + Diagnosis); возвращает результат прогноза

# Авторизация
4. POST /api/auth/login                   — вход администратора; возвращает JWT-токен
5. POST /api/auth/logout                  — выход (инвалидация токена)

# Импорт / Экспорт
9.  POST /api/admin/import/excel          — импорт пациентов из Excel-файла (multipart/form-data)
10. GET  /api/admin/export/excel          — экспорт всех пациентов в Excel-файл (с фильтрами)
```

### Критерии приемки:

```
1. Пациент может пройти опрос до конца и получить результат прогноза на экране.
2. Результаты опроса (Patient + Diagnosis) сохраняются в БД атомарно; при ошибке — полный откат.
3. Форма опросника не отправляется при незаполненных обязательных полях — валидация на фронте и бэке.
4. Администратор получает JWT только при корректной паре логин/пароль; при неверных данных — 401.
5. Эндпоинты /api/admin/* недоступны без валидного JWT — возвращают 403.
6. Импорт Excel: некорректные строки логируются, корректные сохраняются.
7. Экспорт Excel содержит все поля согласно структуре данных ТЗ и корректно открывается в Excel.
8. GET /api/glossaries отдаёт данные из кэша Redis; повторные запросы не инициируют запрос к БД.
9. ML-прогноз возвращается пациенту в рамках запроса POST /api/survey.
10. Покрытие интеграционными тестами (Testcontainers): сохранение опроса, авторизация, импорт/экспорт Excel.
11. Health-эндпоинт /actuator/health возвращает статус приложения и доступность БД и Redis.
```
