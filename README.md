### Ключевые бизнес-правила:
...

*ПРИМЕР:*

```
1. На один `ScheduleSlot` может быть не более одной активной записи.
2. Запись доступна только на будущие слоты со статусом `OPEN`.
3. Врач не может завершить прием, который не относится к нему.
4. После отмены запись не может быть завершена.
5. История изменения статусов приема должна сохраняться.
```

```
1. Пациент не может повторно пройти опрос с тем же `Sys patiend id`
2. Данные опроса сохраняются атомарно: Либо все поля `Patient` и `Diagnosis` записываются, либо ни одно
3. Импорт Excel выполняется транзакционно - при ошибке вся загрузка откатывается или пропускается с логированием
4. Администратор не може получить доступ к панели без действительного JWT-токена
5. Год диагноза не может быь в будущем и не раньше 1900 года
6. Пациент не может отправить форму с незаполненными обязательными полями
```

### Страницы:
```
- Приветственная страница (окошко), выбор `я здесь чтобы пройти опрос` / `я администратор`. Уже после этого переводит на страницу авторизации либо на страницу опросника.
    ├── 4 страницы опросника (возможно изменится)
        └── страница окончания опроса с результатами (вывод => с такой-то вероятностью вы     получите сердечно-сосудистое заболевание ...)
    ├── Страница авторизации
        └── Страница администратора (корректная пара логин-пароль)
            ├── Загрузка данных (импорт из Excel-таблиц)
            └── Выгрузка данных (экспорт в Excel-таблицы)
```

Возможно подключение дашбордов в дальнейшем.

### Обязательные пользовательские сценарии (MVP):
- Пациент проходит опрос и получает результат => результат опроса сохраняется в БД
- Администратор авторизовывается
- Администратор может посмотреть статистику по пациентам и их данные (данные из результатов опроса)
    - (в виде какого-то дашборда, возможно)
- Администратор может импортировать/экспортировать данные в формате Excel-таблиц

### Критерии приемки:
...

*ПРИМЕР:*
```
1. Невозможна двойная запись в один и тот же слот.
2. Ограничения ролей подтверждаются тестами.
3. История статусов приема доступна через API.
```
```
1. Пациент может пройти опрос до конца и получить результат прогноза на экране.
2. Результаты опроса (Patient + Diagnosis) сохраняются в БД атомарно; при ошибке — полный откат.
3. Форма опросника не отправляется при незаполненных обязательных полях — валидация на фронте и бэке.
4. Администратор получает JWT только при корректной паре логин/пароль; при неверных данных — 401.
5. Эндпоинты /api/admin/* недоступны без валидного JWT — возвращают 403.
6. Импорт Excel: некорректные строки логируются, корректные сохраняются.
7. Экспорт Excel содержит все поля согласно структуре данных ТЗ и корректно открывается в Excel.
8. GET /api/glossaries отдаёт данные из кэша Redis; повторные запросы не инициируют запрос к БД.
9. ML-прогноз возвращается пациенту в рамках запроса POST /api/survey.
10. Покрытие интеграционными тестами (Testcontainers): сохранение опроса, авторизация, импорт/экспорт Excel.
11. Health-эндпоинт /actuator/health возвращает статус приложения и доступность БД и Redis.
```
